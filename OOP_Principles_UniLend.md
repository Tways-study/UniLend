# OOP Principles in UniLend

UniLend's `js/app.js` was refactored from a procedural script into a class-based architecture. All four core OOP principles — **Abstraction**, **Encapsulation**, **Inheritance**, and **Polymorphism** — are explicitly demonstrated through six ES6 classes.

## 1. Abstraction

> *Hiding complex implementation details and exposing only a clean, simple interface.*

### `SkeletonLoader`

Before the refactor, skeleton HTML was generated by two loose functions (`skeletonRows`, `skeletonCards`) called throughout the file. Now, callers never need to know the markup — they simply call a static method.

```js
// Caller only sees this:
grid.innerHTML = SkeletonLoader.cards(3);
tbody.innerHTML = SkeletonLoader.rows(5);

// The actual HTML template is hidden inside the class
```

### `Toast`

All DOM construction, Bootstrap wiring, and cleanup logic are hidden inside `Toast.show()`. Any part of the app triggers a notification with one line.

```js
Toast.show("Reservation submitted successfully!", "success");
Toast.show("Cannot approve — item is out of stock.", "error");
```

### `Dashboard` (Abstract Base Class)

`Dashboard` defines a shared interface — `load()` and `_setupLogout()` — without providing a complete implementation. Subclasses are required to fill in the details. If a subclass forgets to override `load()`, the base class throws a descriptive error.

```js
class Dashboard {
  async load() {
    throw new Error(`${this.constructor.name} must implement load()`);
  }
}
```

## 2. Encapsulation

> *Bundling data and the methods that operate on it together, hiding internal state from the outside.*

### `PasswordStrengthMeter`

Before the refactor, the login page held ~35 lines of variables and event listeners inline. Now, all of that is bundled inside one class. The strength configuration, scoring algorithm, and DOM update are all **private** — they cannot be accessed or broken by outside code.

```js
class PasswordStrengthMeter {
  #config = [                        // private: strength labels + colours
    { label: "Weak",   color: "#ef4444" },
    { label: "Fair",   color: "#f59e0b" },
    { label: "Good",   color: "#3b82f6" },
    { label: "Strong", color: "#10b981" },
  ];

  #evaluate(pw) { /* private scoring algorithm */ }
  #update()     { /* private DOM update         */ }

  constructor(passwordInputId, segIds, labelId, wrapId) {
    // wires itself up automatically on instantiation
    this.input.addEventListener("input", () => this.#update());
  }
}
```

Login page usage — no implementation details needed:

```js
new PasswordStrengthMeter("regPassword", ["pwSeg1","pwSeg2","pwSeg3","pwSeg4"], "pwStrengthLabel", "pwStrengthWrap");
```

### `Toast` — Private Static Field

The icon lookup table is a private static field. It is an internal concern of the class and cannot be modified by callers.

```js
class Toast {
  static #iconMap = {
    success: "bi-check-lg",
    error:   "bi-x-lg",
    info:    "bi-info-lg",
  };
}
```

### `StudentDashboard` — Private State + Methods

All mutable state and all helper functions that were previously loose module-level variables are now private class members. Nothing outside the class can touch them.

```js
class StudentDashboard extends Dashboard {
  #allEquipment     = [];   // private: equipment cache
  #filterAvail      = "all"; // private: current filter value
  #equipmentChannel = null; // private: Supabase realtime channel
  #requestsChannel  = null; // private: Supabase realtime channel

  #loadEquipment()       { /* private */ }
  #renderEquipmentCards() { /* private */ }
  #setupSearch()          { /* private */ }
  #loadMyRequests()       { /* private */ }
  #setupReserveForm()     { /* private */ }
  #subscribeToRealtime()  { /* private */ }
}
```

### `AdminDashboard` — Private Methods per Role

Admin-only concerns like the confirm dialog are kept private inside `AdminDashboard`. A student-side caller could never accidentally invoke `#showConfirmModal`.

```js
class AdminDashboard extends Dashboard {
  #inventoryChannel = null;
  #pendingChannel   = null;

  #showConfirmModal({ title, body, okLabel, okColor, onConfirm }) { /* private */ }
  #loadInventory()            { /* private */ }
  #loadPendingRequests()      { /* private */ }
  #loadOverdueReservations()  { /* private */ }
  #setupRestockForm()         { /* private */ }
  #subscribeToRealtime()      { /* private */ }
}
```

## 3. Inheritance

> *A subclass acquires the properties and behaviours of its parent class, avoiding duplication.*

`StudentDashboard` and `AdminDashboard` both extend `Dashboard`. Neither subclass needs to redeclare the Supabase client, the current user, or the logout logic — they inherit all of it for free.

```
Dashboard
├── constructor(supabaseClient, user)  ← inherited by both subclasses
├── load()                            ← overridden by both subclasses
└── _setupLogout(channels)            ← inherited and called by both subclasses
     │
     ├── StudentDashboard extends Dashboard
     │     - inherits: this.db, this.user, _setupLogout()
     │     - adds:     #allEquipment, #filterAvail, channels
     │     - overrides: load()
     │
     └── AdminDashboard extends Dashboard
           - inherits: this.db, this.user, _setupLogout()
           - adds:     #inventoryChannel, #pendingChannel
           - overrides: load()
```

Both subclasses call `super(supabaseClient, user)` in their constructors, delegating base initialisation upward:

```js
class StudentDashboard extends Dashboard {
  constructor(supabaseClient, user) {
    super(supabaseClient, user); // inherits this.db and this.user
  }
}

class AdminDashboard extends Dashboard {
  constructor(supabaseClient, user) {
    super(supabaseClient, user); // same
  }
}
```

The shared `_setupLogout()` method in the base class is called by both subclasses at the end of their realtime setup. Neither subclass contains duplicate logout code.

```js
// In StudentDashboard:
this._setupLogout([this.#equipmentChannel, this.#requestsChannel]);

// In AdminDashboard:
this._setupLogout([this.#inventoryChannel, this.#pendingChannel]);
```

## 4. Polymorphism

> *The same interface produces different behaviour depending on the object it is called on.*

### Method Overriding — `load()`

The page boot code calls the exact same method on both dashboard types. The runtime dispatches the correct role-specific sequence automatically.

```js
// Student page:
const dashboard = new StudentDashboard(supabase, session.user);
await dashboard.load(); // → greeting, equipment cards, my requests, search, reserve form, realtime

// Admin page:
const dashboard = new AdminDashboard(supabase, session.user);
await dashboard.load(); // → inventory table, pending requests, overdue list, restock form, realtime
```

Both call sites are identical — `new XxxDashboard(...); await dashboard.load()` — but what `load()` actually does depends entirely on which subclass was instantiated. This is **runtime polymorphism**.

### `Toast.show()` — Type-driven Output

The same static method call produces a visually distinct notification based on the `type` argument. Different icons, colours, and CSS classes are applied internally.

```js
Toast.show("Request approved.",               "success"); // green  + check icon
Toast.show("Failed to approve request.",      "error");   // red    + X icon
Toast.show("Request denied.",                 "info");    // neutral + info icon
```

### Status Badge Rendering — Data-driven Polymorphism

Inside `#loadMyRequests()`, the same rendering path produces a different badge for every possible reservation status. The `displayStatus` variable drives which CSS class and icon is applied — same code, different output.

```js
const icon = isOverdue               ? "bi-exclamation-circle-fill"
           : r.status === "approved" ? "bi-check-circle-fill"
           : r.status === "returned" ? "bi-arrow-return-left"
           : r.status === "denied"   ? "bi-x-circle-fill"
           : "bi-clock-fill";                               // pending
```

## 5. Class Overview

| Class | Extends | Principle(s) |
|---|---|---|
| `SkeletonLoader` | — | Abstraction |
| `Toast` | — | Abstraction, Encapsulation (`#iconMap`), Polymorphism (`type`) |
| `PasswordStrengthMeter` | — | Encapsulation (`#config`, `#evaluate`, `#update`) |
| `Dashboard` | — | Abstraction (shared interface), Inheritance (base class) |
| `StudentDashboard` | `Dashboard` | Inheritance, Encapsulation (`#` fields + methods), Polymorphism (overrides `load()`) |
| `AdminDashboard` | `Dashboard` | Inheritance, Encapsulation (`#` fields + methods), Polymorphism (overrides `load()`) |

## Key Takeaways

- **Abstraction** reduces cognitive load — callers work with `SkeletonLoader.cards()`, `Toast.show()`, and `dashboard.load()` without knowing the DOM mechanics.
- **Encapsulation** protects state — private fields (`#`) ensure that equipment caches, channels, and configuration cannot be accidentally mutated from outside the class.
- **Inheritance** eliminates duplication — the Supabase client, user object, and logout handler exist once in `Dashboard` and are shared across both role dashboards.
- **Polymorphism** keeps boot code clean — the same two-line instantiation pattern works for both roles; the correct behaviour is resolved at runtime by the class hierarchy.
